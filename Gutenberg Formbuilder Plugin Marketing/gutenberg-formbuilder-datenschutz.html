Datenhoheit & revDSG · Gutenberg Formbuilder

Ihre Daten verlassen die Site nicht.
Auch nicht kurz für die Verarbeitung.

Externe Form-Anbieter sind bequem – bis zur ersten Datenschutzprüfung. Ab dann zählen AVV, Drittlandtransfer und Verarbeitungsverzeichnis. Wer Formulardaten auf der eigenen Site verarbeitet, spart sich diese Fragen von Anfang an.

Externer Form-Anbieter
  • Auftragsverarbeitungsvertrag (AVV) mit dem Anbieter nötig
  • Datentransfer in Drittland prüfen (USA, UK usw.)
  • Cookie-Banner für Anbieter-Skripte anpassen
  • Eintrag ins Verarbeitungsverzeichnis pflegen
  • Auskunftsbegehren beim Anbieter anfragen
  • Anbieter kann AGB und Datenlage ändern
Gutenberg Formbuilder
  • Kein AVV mit Dritten – Sie sind Verantwortlicher, fertig
  • Kein Drittlandtransfer – Daten bleiben auf Ihrem Server
  • Keine Anbieter-Skripte – kein Cookie-Banner nötig
  • Verarbeitungsverzeichnis: nur Ihr eigener WordPress-Server
  • Auskunftsbegehren direkt im WordPress-Backend bearbeiten
  • Open Source – keine AGBs, kein Anbieter-Risiko
Wie es technisch funktioniert

Daten direkt in Ihrer WordPress-Datenbank – nirgendwo sonst.

Das Plugin schreibt jede Formulareinsendung in dieselbe Datenbank, die Ihre Site ohnehin betreibt. Kein Zwischenstopp, kein externer API-Call, kein Drittanbieter-Speicher.

1

Speicherung direkt in WordPress

Jede Formulareinsendung landet in der Tabelle wp_gfb_submissions in Ihrer eigenen WordPress-Datenbank – auf Ihrem Server, unter Ihrer Kontrolle. Kein Request geht an einen externen Dienst. Felder, die Sie als «vertraulich» markiert haben, werden vor dem Speichern mit AES-256-GCM verschlüsselt; der Schlüssel liegt in Ihrer wp-config.php, nicht in der Datenbank selbst.

Resultat: Ein Datenbankdump ohne den Schlüssel enthält keine lesbaren vertraulichen Daten.
2

Auskunft & Löschung – WordPress-native

Das Plugin hängt sich in die WordPress-Hooks personal_data_exporter und personal_data_eraser ein. Ein Auskunftsbegehren nach revDSG Art. 25 bearbeiten Sie direkt unter «Werkzeuge → Datenschutz» in WordPress: alle Formulardaten der betreffenden Person werden aufbereitet und als ZIP-Export bereitgestellt – inklusive aller Felder, verschlüsselte Felder entschlüsselt für die Auskunft. Dasselbe gilt für Löschanfragen: der Eraser entfernt alle Einträge, die mit der angefragten E-Mail-Adresse verknüpft sind.

Kein Anruf beim Anbieter, kein API-Key, kein Warten – alles im eigenen Backend.
3

IP-Pseudonymisierung über Filter

Das Plugin speichert standardmässig die vollständige IP-Adresse der einsendenden Person – für Debugging und Rate-Limiting. Wer das aus Datenschutzgründen einschränken möchte, hängt sich in den Filter gfb_pseudonymize_ip ein. Damit wird die IP vor der Speicherung gekürzt (IPv4: letztes Oktett entfernt; IPv6: letzte 80 Bit auf null gesetzt). Die Änderung braucht zwei Zeilen Code im Theme oder einem eigenen Mini-Plugin.

Beispiel: add_filter('gfb_pseudonymize_ip', '__return_true');
4

Keine Drittanbieter-Skripte im Frontend

Das Plugin lädt im Frontend ausschliesslich eigene Assets – kein Google Fonts vom CDN, kein reCAPTCHA, kein Analytics-Pixel, kein externer Skript-Tag. Was die Besucherin in Ihrem Formular sieht und was ihr Browser lädt, stammt von Ihrem Server. Das bedeutet: kein neuer Cookie-Banner-Eintrag, keine zusätzliche Einwilligung für Drittdienste.

Spam-Schutz ohne externes Captcha

Drei Schutzmechanismen – kein reCAPTCHA, kein Drittdienst.

reCAPTCHA schickt Nutzerdaten an Google. Das Plugin schützt Formulare ohne diesen Umweg – und ohne dass Ihre Besucherinnen Ampeln oder Fussgängerstreifen anklicken müssen.

🪤

Honeypot pro Formular-Instanz

Jedes Formular enthält ein verstecktes Feld, das für Menschen unsichtbar ist. Bots füllen es aus – und das Plugin verwirft die Einsendung still. Der Honeypot-Feldname wechselt pro Formular-Instanz, damit einfache Bot-Listen nicht greifen.

🔏

HMAC-Anti-Replay-Token

Beim Laden des Formulars erzeugt das Plugin einen HMAC-signierten Einmal-Token mit Ablaufzeit. Die Einsendung ist nur gültig, wenn der Token stimmt und noch nicht verwendet wurde. Mehrfachversand desselben Formulars – ob durch Bot oder manuelles Reload – wird abgewiesen.

⏱

Rate-Limit

Pro IP-Adresse lässt das Plugin innerhalb eines konfigurierbaren Zeitfensters nur eine bestimmte Anzahl Einsendungen durch. Das Limit gilt pro Formular, nicht global. Wer das Limit erreicht, erhält eine neutrale Fehlermeldung – kein Hinweis darauf, dass ein Rate-Limit aktiv ist.

Was wir nicht versprechen

Konformität ist mehr als ein Plugin.

Das Plugin nimmt Ihnen technische Arbeit ab. Es ersetzt keine rechtliche Beratung und deckt nicht alle Pflichten des revDSG allein ab.

Keine automatische revDSG-Konformität

Ob Ihr Betrieb konform ist, hängt von mehr ab als einem Plugin: Verarbeitungsverzeichnis, Datenschutzerklärung, interne Prozesse, Mitarbeiterschulungen. Das Plugin liefert die technische Grundlage – den Rest müssen Sie oder Ihre Rechtsberatung sicherstellen.

Kein Ersatz für einen AVV-Prozess

Wenn Sie WordPress bei einem Hosting-Anbieter betreiben, sind Sie mit diesem in einer Auftragsverarbeitungsbeziehung. Das Plugin ändert daran nichts – aber es reduziert die Zahl der AVVs auf diesen einen, unvermeidlichen.

Kein Rechtsgutachten

Die Einschätzungen auf dieser Seite sind technischer Natur und dienen der Information. Sie ersetzen keine Beratung durch eine auf Datenschutzrecht spezialisierte Person.
