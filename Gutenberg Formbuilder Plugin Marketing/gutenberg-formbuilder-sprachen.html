Eine Schweizer Site bedient oft drei Landessprachen. Plugins, die nur Englisch sprechen, zwingen zur manuellen Strings-Übersetzung oder zu kostenpflichtigen Add-ons. Der Gutenberg Formbuilder liefert fertige Sprachdateien – ohne Aufpreis, ohne externe Dienste.

Die vier mitgelieferten Sprachen

Frontend und Admin – alles übersetzt.

Alle Strings, die Besucherinnen sehen (Fehlermeldungen, Erfolgstext, Validierung) und alle Admin-Labels sind in vier Sprachen verfügbar. WordPress lädt automatisch die passende Datei – abhängig von der Spracheinstellung der Site.

Wie es funktioniert

WordPress wählt die Sprache – das Plugin folgt.

Es gibt keine eigene Spracheinstellung im Plugin. Die Sprache kommt aus WordPress – das ist der richtige Weg. Keine doppelte Konfiguration, keine Inkonsistenz.

1

Quellsprache Deutsch

Alle Strings im Code sind auf Deutsch verfasst. Das ist die Quellsprache des Plugins. Wer eine deutschsprachige Site betreibt, braucht keine Sprachdatei – die Texte kommen direkt aus dem Code.

2

Locale folgt den WordPress-Einstellungen

WordPress liest die Sprache aus «Einstellungen → Allgemein → Sprache der Website». Das Plugin registriert seine Sprachdateien via load_plugin_textdomain() – WordPress lädt daraufhin automatisch die passende .mo-Datei aus dem Ordner languages/.

Beispiel: Site-Sprache «Français» → Datei gutenberg-formbuilder-fr_FR.mo wird geladen.

3

Mehrsprachige Plugins setzen den locale-Filter

Plugins wie Polylang oder WPML können den locale-Filter pro Seite oder Benutzer setzen. Der Formbuilder reagiert auf diesen Filter – auf einer französischsprachigen Seite sieht die Besucherin Formulartext auf Französisch, auch wenn die Admin-Sprache Deutsch ist.

4

Datum und Uhrzeit folgen dem WordPress-Format

Platzhalter wie tt.mm.jjjj oder hh:mm werden nicht hartcodiert. Das Plugin liest das Datumsformat aus den WordPress-Einstellungen («Einstellungen → Allgemein → Datumsformat») und zeigt den Platzhalter passend an.

Wer F j, Y als Format eingestellt hat, bekommt auch im Formular den entsprechenden Platzhalter.

Kein manuelles Eingreifen nötig. Sobald die Site-Sprache in WordPress gesetzt ist, wechselt das Plugin mit. Sprachübersetzungen werden nicht im Plugin gespeichert, sondern von WordPress verwaltet – so wie es der Standard vorschreibt.
Ehrliche Grenzen

Was noch fehlt.

Vier Sprachen sind ein guter Anfang für die Schweiz. Zwei Lücken bleiben offen – wir benennen sie, bevor Sie danach fragen.

Rumantsch

Die vierte Landessprache fehlt noch. Eine rm_CH-Sprachdatei ist in Planung. Wer eine rätoromanische Site betreibt und übersetzen möchte, kann sich per GitHub-Issue melden – wir koordinieren gerne.

Right-to-left-Sprachen

Arabisch, Hebräisch und andere RTL-Sprachen funktionieren technisch (WordPress liefert das dir="rtl"-Attribut), aber das CSS wurde nicht systematisch auf RTL-Layouts getestet. Optisch kann es zu Verschiebungen kommen.

Open Source · Beitragen

Übersetzung fehlt? Sie können sie beisteuern.

Das Plugin ist Open Source. Die Sprachdateien liegen offen im Repository – wer eine Sprache hinzufügen oder eine bestehende korrigieren möchte, ist willkommen.

📂

PO- und MO-Dateien im Repo

Im Ordner languages/ liegen die editierbaren .po-Dateien (lesbar mit Poedit oder jedem Texteditor) und die kompilierten .mo-Binärdateien. Die Struktur folgt dem WordPress-Standard.

🔁

Pull Request auf GitHub

Eine neue Sprachdatei einreichen: .po und .mo im richtigen Locale-Format (gutenberg-formbuilder-[locale].po) als Pull Request auf github.com/BlitzDonner/gutenberg-formbuilder. Wir prüfen und mergen.

💬

Rumantsch auf Anfrage

Wer eine rätoromanische Übersetzung benötigt und Ressourcen hat, sie zu erstellen: ein GitHub-Issue genügt. Wir erstellen die Vorlage aus der deutschen Quelldatei und koordinieren den Review.
